## What does NOT get committed

(Already in `.gitignore`; listed here for clarity.)

```
snapshots/       working data, regenerated by the harvest
exports/         working data, regenerated from the DB
reports/         working data, regenerated by export + benchmark scripts
logs/            harvest/enrichment runtime logs
bibliography_browser.html   regenerated artifact
__pycache__/     Python bytecode cache
```

These directories will be created on first run, but they should never be committed. The `.gitignore` enforces this.

## One-time repo creation steps

1. **Create the public repository** on GitHub (or GitLab / Codeberg / wherever you host).
   Suggested name: `tnbc-atlas-pipeline` under the project organization.
   Description: "Bibliography harvest, dedup, enrichment, and tier-assignment pipeline for TNBC Atlas (tnbc.info)."

2. **Initialize from this directory**:
   ```bash
   cd "AI ML Lab/tnbc_atlas_pilot"
   git init
   git add LICENSE README.md CONTRIBUTING.md PUBLISH.md Makefile requirements.txt .gitignore
   git add sql/ scripts/ curation/ docs/ .github/
   git commit -m "Initial public release of the TNBC Atlas pipeline"
   git remote add origin https://github.com/<org>/tnbc-atlas-pipeline.git
   git branch -M main
   git push -u origin main
   ```

3. **Configure repository secrets** at GitHub → Settings → Secrets and variables → Actions:
   - `SUPABASE_DATABASE_URL` — pooled URI from Supabase Settings → Database
   - `CONTACT_EMAIL` — project contact email for upstream API polite-pool headers
   - `R2_ACCOUNT_ID`, `R2_ACCESS_KEY`, `R2_SECRET_KEY` — for nightly export uploads to Cloudflare R2

4. **Trigger the first workflow manually**: Actions tab → Weekly harvest → Run workflow.
   Confirms secrets and Supabase connectivity before any scheduled run fires.

3. **Verify the `.gitignore` is working** &mdash; `git status` should show no untracked files in `snapshots/`, `exports/`, `reports/`, or `logs/`.

4. **Add a small README badge block** linking to the live website (tnbc.info), the methods page, and the relevant CI status once CI is wired up.

## What this repo does not include

- Database backups or production data dumps. (Operationally separate; not appropriate for a public code repo.)
- Cloudflare deployment configuration. (Lives in the website repo.)
- The editorial board's review notes on tier-1 / tier-2 promotion. (Editorial workflow lives in a private project space.)
- Secrets, API keys, connection strings. (Use environment variables; never commit secrets.)

## Maintenance

Once published, contributions follow the workflow in `CONTRIBUTING.md`. The maintainer team owns merges to `main`; tagged releases (e.g. `v0.2`, `v1.0`) mark stable points the methods page can reference by version.
